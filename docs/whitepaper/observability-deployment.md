# Observability & Deployment Model

Full visibility into the distributed microservice architecture is achieved through a unified observability stack, natively integrated into the Kubernetes deployment model.

## 1. Observability Stack

The platform utilizes three core pillars of observability: Metrics, Tracing, and Dashboards.

### Prometheus (Metrics)
Standardized metrics are generated by every FastAPI microservice using a shared, lightweight `ServiceMetrics` engine.
- **Scraping Model:** Prometheus auto-discovers targets using Kubernetes Pod annotations (`prometheus.io/scrape: "true"`).
- **Standard Metrics:** Every service tracks HTTP `request_total`, `error_total`, and `request_duration_seconds` (Histogram).
- **Domain Metrics:** Custom histograms track specific ML prediction times, token generation latency, and DB vector search durations.

### Jaeger via OpenTelemetry (Distributed Tracing)
To debug performance bottlenecks across multiple network hops, the system implements W3C Trace Context propagation.
- **Headers:** `traceparent` headers are injected and parsed at every service boundary.
- **Hierarchy:** Spans are organized into Root → Child relationships (e.g., `Gateway` → `RAG` → `DB Query`).
- **Error Capture:** Python exceptions caught within a span's context manager automatically append error attributes (`error.type`, `error.message`) to the trace without dropping the data point.

### Grafana (Dashboards)
Grafana automatically provisions the Prometheus datasource via Kubernetes ConfigMaps and visualizes:
1. **Service Health:** Error rates and active circuit breakers.
2. **Latency Distributions:** P50/P95/P99 latency heatmaps for ML inference and LLM generation.
3. **Infrastructure:** Pod CPU/Memory utilization to validate HPA scaling actions.

## 2. Kubernetes Deployment Model

Biz Stratosphere is deployed using Kubernetes manifests, orchestrated via Kustomize for single-command (`kubectl apply -k .`) execution.

### Stateless Application Tiers
The five Python microservices (`api-gateway`, `ml-inference`, `rag-service`, `llm-orchestrator`, `embedding-worker`) are deployed as standard `Deployments`.
- **Strategy:** `RollingUpdate` with `maxUnavailable: 0` ensures zero-downtime redeployments.
- **Probes:** `livenessProbe` (`/health`) and `readinessProbe` (`/ready`) ensure traffic is only routed to fully initialized Python processes.

### Stateful Data Tiers
Data persistence and high-compute singletons require specialized controllers.
- **PostgreSQL:** Deployed as a `StatefulSet` with an attached `PersistentVolumeClaim` (PVC). This guarantees stable network identifiers and ensures vector embeddings survive pod restarts.
- **Ollama:** Deployed as a `Deployment` but uses a `Recreate` strategy. Because LLM models require exclusive access to host memory/GPU resources during loading, concurrent rolling updates are disallowed to prevent Out-Of-Memory (OOM) kernel panics.

### Networking & Service Discovery
- **Internal:** Inter-service communication relies exclusively on Kubernetes CoreDNS. Services interact via stable addresses (e.g., `http://ml-inference:8001`).
- **External:** External ingress is handled by the NGINX Ingress Controller, which applies host-based routing to direct traffic to the `api-gateway`, `grafana`, or `jaeger` UIs. ConfigMap-based environment variables inject these URLs into the application pods at startup.
